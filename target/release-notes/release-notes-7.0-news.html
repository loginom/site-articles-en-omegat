Large-Scale Update of Loginom Analytical Platform Loginom became cross-platform. Standard and Enterprise editions are available for Linux and Windows operating systems. Integration with Kafka message broker became possible. New features appeared in JavaScript handler when importing data from files as well as in OLAP cube visualizer and Reports. Databases operation was improved and optimised. Performance was improved.

When preparing this release, the main efforts were focused on Loginom migration to Linux operating system. The changes affected mostly the technologic platform stack and they are connected with system challenges and libraries, third-party libraries, operations with rows and different integrations. At the same time, 80% percent of the server part was transferred from the previous version, and the client part was not significantly changed. That is why the workflows prepared in Loginom for Windows will be used with Linux as well. In some cases only reconfiguration of nodes will be required.

The most important changes are described below.

Loginom Restrictions for Linux

  1. There is no integration for 1С:Enterprise via COM connection because COM server is available only for Windows. To provide integration with 1С:Enterprise, it is possible to use alternative mechanisms, for example, HTTP protocol.
  2. It is not possible to connect to MS Excel as to database because there is no software driver for Linux. Only xlsx format is supported (MS Excel 2007 and higher).
  3. There is no built-in integration with MS Access because open source driver is not available.
  4. There is no export to Tableau.
  5. There is no Program execution handler that is used to start the external Windows programs.

Optional Loginom Features for Linux

Logging in journald is available in Linux. Corresponding configurable parameter is provided on Loginom Administration page. systemd.so.0 library is required to use the logging method (it is available for sure if systemd initialization system is used in Linux distribution kit).

Integration with Kafka Message Broker

Kafka Apache is a popular distributed system of message exchange between server applications in real time. Due to its high capacity, scalability and failure safety it is applied in large-scale projects with large volumes of data.

Import and export of messages to Kafka queue became possible in Loginom Enterprise. Integration with Kafka is provided according to the same principle as integration with databases. Connection is configured at the first stage, and then the nodes based on Kafka import or export components are used.

Database Use Improvements

Considerable work aimed at improvement and optimisation of database use has been done. The following support has been provided:

  • SCRAM-SHA-256 Authentication and SCRAM-SHA-256-PLUS Authentication for PostgreSQL;
  • SHA-256 authentication for MySQL 8 when connecting without client use;
  • connection to Oracle 18c иand higher without client use;
  • Microsoft OLE DB Driver client for SQL Server for connection to MS SQL Server.

SQLite database can be used in-memory now. When configuring connection to SQLite database, it is sufficient to select corresponding checkbox and it will be expanded in memory. Data processing speed will be considerably increased in such case of use. While DB is in memory, it is possible to connect to it from another package. Still there is access rights differentiation to DB SQLite in-memory. Database is kept in memory until its connection is active.

It became possible to install inbuilt Firebird client.

Updated import from database wizard

Import from database wizard was completely modified. There was not enough space for the request text in the SQL request mode in the previous version, and it was also impossible to see variables and use them. It became much more comfortable to work with SQL requests in the updated wizard:

  • it became possible to select the "Table selection" or "SQL request" wizard representation mode;
  • block with a list of variables was added in the SQL request mode;
  • it is possible to drag tables, fields and variables to drag&drop SQL code editor;
  • variable value can be inserted into SQL code editor as parameter (by default) or as macro substitution;
  • field filtration functionality is available in the SQL request mode in the DB object tree (when tree is expanded);
  • it became possible to set the table name via variables;
  • loading of DB objects to the tree of objects became faster.

Import from Files

It became possible to read data importing from files (txt, csv, XML, Excel) having executed http request beforehand. For this purpose, URL can be specified instead of the file name in import from file wizard. HTTP Basic Authentication is supported. It means that it is possible to transfer login and password in URL. When importing txt, csv, xml formats for preview or format estimate, only the required data volume is downloaded. When it comes to xlsx format, the whole file is downloaded to memory that is why it is required to pay attention to size of such files.

Web Services Operation

REST and SOAP request handlers function without changes. However, when transferring workflows between Loginom for Windows and Linux, in some cases it will be required to reconfigure REST connections and SOAP service. Authentification via client certificate is not supported in the Linux version. In Windows it was selected from the storage of personal certificates. When working with Linux, it is required to set the certificate file, key and password.

The default value of "Allowable Content-type of response" parameter was changed in the REST service connection wizard. Instead of */* application/json is installed.

The output data set will be changed for the packages created before version 6.4.0 in the "Additional data" port of the REST service node. Previously, the value in "Error code" column was equal to 0 irrespective of HTTP Status Code. Now, if HTTP Status Code >= 400, Error Code will be equal to 6.

Programming languages use

Now JavaScript handler includes File Storage API that enables to execute all file operations without rights setup and in a safe manner:

  • Get file list.
  • Create, delete, move, rename folders.
  • Delete, move, rename files.
  • Check path correctness.

It became possible to start Python in a separate process, as well as simultaneously. Start both within the Loginom process and in a separate process is available in Windows editions, whereas operation in Linux is supported only in a separate process.

When sending pandas tables in output_table for column captions in Python, transliteration in prepare_compatible_table function is not performed. This is especially true if captions of pandas columns are written in Cyrillic characters.

Processing Components Improvements

  1. When entering data from the keyboard in "Date/time" editors, it is possible to specify only date without time. Previously, it was necessarily required to set time.
  2. Optional input table port appeared in Supernode handler. It enables to make design of workflows more flexible. Port optionality checkbox was added in the Supernode wizard. It is possible to edit the checkbox only for the table port. When collapsing nodes to theSupernode, optionality checkbox is selected only for the input ports related to the Table type to which optional ports are connected inside the Supernode.
  3. Functions of base64 encoded rows transformation to binary data rows and vice versa are available in Calculator in the JavaScript mode.
  4. Data caching option was added in Sorting handler.
  5. "Replace with set value" processing method became available for all data types in Eliminate outliers. Previously, it was possible to use it only for the string and logical types.
  6. It is allowed to introduce changes compatible with earlier configured lists of selected elements into the XSD schema of request and response. Now such changes do not cause "Input XSD schema does not comply with the node settings" error when activating nodes. The changes concern all nodes in the process of which generation or parsing of xml document is performed according to the schema: XML file import, Export to XML, XML extraction, XML generation, SOAP request.

Ease of Use and UI/UX Improvements

  • Loginom login and password can be transferred in URL now.
  • It became possible to add workflow notes. Now it became possible to document workflow directly during its designing by adding text blocks in any place. Pre-installed set of color styles is provided for notes blocks.
  • It became possible to configure dimensions in a separate window in Cube visualizer. The wizard is called using the button from the Cube Toolbar and corresponding context menu item. Previously, after adding each dimension, the Cube was rebuilt, and the Cube configuration took much time for the large data volumes. Cube configuration process became much more comfortable and faster now.
  • It became possible to create subgroups of any nesting level in groups on Reports page.
  • On Package variables page Navigation bar became available.
  • "Copy full path" item was added to the context menu of the file storage manager. When pressing this item, names of the selected files/folders are copied to the clipboard in the order in which they are represented in the table. It is possible to copy to the clipboard only when connecting via https protocol.
  • The default action is in bold in all context menus of application.
  • Text of the message that is displayed during the long visualizer preparation was changed (for example, when entering Statistics or Cube visualizer when calculations are performed at the moment of activation). Now "Node is locked" is displayed instead of "Loading".
  • Tree nodes icons were removed in the Property Inspector. It enabled to display the property names more fully under conditions of limited width of the Property Inspector window.

Data Handling

"Compare string unique values in a binary way" option appeared in the package properties. String comparison will be changed after closing the package and all opened packages dependent on it. When changing "Compare string unique values in a binary way" setting, retraining of trained nodes may be required. The option effect extends to the following components:

  • Clustering
  • EM Clustering
  • Self-Organizing Network
  • Neural Network (Classification)
  • Neural Network (Regression)
  • Linear Regression
  • Logistic Regression
  • ARIMAX
  • Coarse Classes
  • Sampling (sampling with bias)
  • Partitioning (sampling with bias),

and also to the following visualizers:

  • Binary Classification Assessment
  • Linear Regression Report
  • Logistic Regression Report

Value of "Compare string unique values in a binary way" option is displayed in the Property Inspector.

Now it is possible to install Random Seed.in the wizard of the handlers algorithms of which use generation of pseudorandom numbers. Random Seed value can be set via variable, it is also possible to generate or select "Always randomly" checkbox. Command to copy value to the clipboard became available.

Changes in Loginom Integrator

Loginom Integrator configuration file was changed:

  • web.config file name was replaced with Integrator.dll.config.
  • appSettings, system.web, system.serviceModel and system.webServer elements were deleted with all child elements.

  • configSections element format was changed:

    before:

      <configSections>
        <section name="log" type="DeductorIntegrationServer.Core.LogConfigurationSection"/>
         <section name="loginom" type="DeductorIntegrationServer.Core.DeductorConfigurationSection"/>
  </configSections>

    after:

      <configSections>
        <section name="log" type="Integrator.Core.LogConfigurationSection, IntegratorCore"/>
        <section name="loginom" type="Integrator.Core.LoginomConfigurationSection, IntegratorCore"/>
  </configSections>

  • For loginom element optional urlPathPrefix attribute was added. Its value is used as prefix to address of all REST and SOAP end points. An empty string is the default value of this attribute. For example, if Loginom Integrator is used at the following address: http://localhost/lgi, and urlPathPrefix attribute is not specified, help page for REST is available at the following address: http://localhost/lgi/rest/help. If urlPathPrefix="abc", address was changed as follows: http://localhost/lgi/abc/rest/help. To use the same addresses in the new Integrator as in the old one, it is required to specify as follows: urlPathPrefix="Service.svc".

  • For loginom element optional swaggerUI attribute was added. It can take true or false value. The attribute enables/disables generation of openapi/index.html page that implements access to the published services via Swagger UI.

It became possible to set location of Loginom Integrator configuration file. Location of the folder from which Integrator.dll.config and appsettings.json configuration files are downloaded is defined according to the following rules:

  • if configdir parameter is specified in the command line, the specified folder is used (the parameter name can be specified in any register;
  • if the parameter is not specified and in the current folder ($PWD) there is Integrator.dll.config file, the current folder is used;
  • otherwise the folder with Integrator.dll file is used.

If there is no Integrator.dll.config file in the specified folder, Integrator is not started, and the user receives an error message.

The relative path to the work Integrator directory is defined relative to the configuration file. If in Integrator.dll.config file the value of workDir attribute is the relative path, it is interpreted as the path relative to Integrator.dll.config file. Value ofworkDir attribute can be equal to . (current folder). It will mean that workdir matches configdir.

A separate wsdl file is generated for each published package (the earlier description of the SOAP service included content of all published packages). Generation of root wsdl document that refers to all other wsdl documents was added. By default, root wsdl is generated at the following address:/soap?wsdl. If the value of urlPathPrefix="Service.svc" attribute is set in the Loginom configuration file, wsdl is additionally generated by the following address: /Service.svc?wsdl that is used in the old Integrator.

Now OpenAPI and Swagger UI are supported in Loginom Integrator. Now it is possible to execute requests to test API on the following page: http://host/lgi/openapi/index.html.

Now it is possible to connect to Loginom Server via Unix domain socket. For this purpose, for server element in the Loginom Integrator configuration file unixsocket attribute must be added. Its value matches the value of "Unix socket" parameter set in the connection parameters on "Administration" page in Loginom Server.

When installing Loginom Integrator, it is possible to set Unix Socket parameter in the customization dialog. The default value of this parameter for all editions is %ProgramData%\Loginom\Server\loginomd.socket.

Algorithm of Loginom Integrator connection to Loginom Server is described below:

  • if unixsocket and host attributes are specified in the Integrator configuration file, first, an attempt to establish connection to unix socket is made, and if that failed, TCP is used for connection;
  • if no server elemet is specified in the configuration file, first, an attempt to establish connection to unix socket is made, and then TCP is used for connection;
  • connection only to unix socket is used in Integrator Team.

If Loginom Server is not on localhost relative to Integrator, when installing Loginom Integrator, Unix Socket parameter value set by default must be deleted (leave the field empty) to avoid unnecessary connection attempts.

Logged server connection/disconnection messages, etc. are generated taking into account the protocol via which connection is established.

Now it is possible to create unix socket in Loginom Server Team if there is no Bindings element in the configuration file. Socket file is created in the work Loginom Server directory, and it has loginomd.socket name.

Connection via Unix domain socket when using docker containers eliminates the need to create the virtual network between containers. Besides, performance is higher when connecting via UDS both for Linux, and for Windows.

Processing of X-Forwarded-Proto and X-Forwarded-Host headers was added. These headers influence the generated URLs in OpenAPI documents and on help page for REST service.

Format of response SOAP messages was hanged for old packages created before version 6.3.0. The same message format as for the new packages will be used for the old ones . The user will receive corresponding message when converting.

It became possible to output Loginom Integrator logs to journald for the Linux version. <journald level="Off" maxEntrySize="65536"/> element was added to the configuration file. It configures logging in journald for Linux. This element is ignored for Windows. maxEntrySize attribute sets the maximum message size in bytes. If there is no attribute or if it contains the negative value, the following default value is used: 2147483647.

It became possible to output Loginom Integrator logs to the console. In log section of the configuration file optional console element was added. The level of logging to the console is configured in it. The same message format as for logging to file is used by default. But if DOTNET_RUNNING_IN_CONTAINER environment variable is set with true or 1 value (namely, Integrator was started from Docker container), the current date/time is not recorded in the logged messages. Logs with Warn level and higher are written in stderr, whereas the other ones in stdout. If logs are redirected from the container console to host journald, the multiline messages are divided into several ones (each line becomes a separate message in journald).

Error message format was changed in Loginom Integrator. The format that matches RFC 7807 was implemented for RESR responses. JSON with error description is returned for the errors with code 400 in REST service. Some changes were introduced into the element structure for SOAP services.

Performance

The speed of csv files import and export was increased by 1.3 times due to transition from UTF16 to UTF-8.

xml files are faster processed due to transition to new libraries. For example, when converting the package from the old version to the new one, performance is increased by 1.3 times.

REST client operation speed was increased particularly when web service was accessed many times in one node. In some cases it occurred up to 5 times faster.

Other Changes

Please note!

  1. The mechanism of row comparison and conversion was changed in Loginom 7. ICU library — International Components for Unicode is used instead of built-in Windows functions. It provides the same results of work with rows on different platforms. The order of row sorting can be changed in the workflows prepared in the earlier Loginom versions.
  2. Algorithm of pseudorandom numbers generation was changed. In this regard, Sampling and Partitioning can generate another partitioning. "Imputation with random values" method works differently in Imputation handler. "Tile" partitioning method with "Set randomly" type of matching observations processing works differently in Binning.
  3. Internet Explorer browser is not supported.
  4. The 32-bit version of Windows operating systems is not supported. Minimum requirements: Windows 10, Windows Server 2019.
